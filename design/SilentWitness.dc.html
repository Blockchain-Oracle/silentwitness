Getting started / Introduction

SilentWitness

A hypothesis-first AI incident-response investigator for the SANS SIFT Workstation. Point it at a Windows disk or memory image; it forms forensic hypotheses, pulls evidence from a parsed index, and produces a cited, defensible report — with architectural guardrails that structurally prevent it from hallucinating or modifying evidence.

New here

Go from a blank SIFT VM to a finished report in five commands — start with the Quickstart.

The wedge

Most AI-DFIR tools let the model free-read raw evidence and trust its output. SilentWitness inverts that. The agent's only interface is a typed search over pre-parsed evidence, every claim must quote a real record (verified server-side), and a framework-level gate refuses to let it conclude until all five investigative questions are answered. The guardrails are architectural, not prompt-based — enforced at the server and filesystem boundary, so they hold even when the model ignores its instructions.

Facts you can quote

  • A custom MCP server exposing 27 typed tools; model-agnostic across OpenAI, Anthropic, Google, and Ollama.
  • Evidence index built from 9 targeted parsers plus a Sigma detection engine (pySigma).
  • 100% recall (10/10) on the real SANS ROCBA case in the headline run — with the full, honest distribution published.
  • Every finding is traceable to the exact tool execution that produced it. MIT licensed; runs on SIFT 2026.

Getting started / Quickstart

Quickstart

On a SANS SIFT 2026 workstation with your own LLM API key, five commands take you from raw image to scored report.

sift@workstation: ~
# 1 · install (bootstraps uv + every forensic dependency)
curl -fsSL https://raw.githubusercontent.com/Blockchain-Oracle/silentwitness/main/install.sh | bash

# 2 · pick a model — anthropic / openai / google / ollama
export SILENTWITNESS_MODEL=openai:gpt-5.5
export OPENAI_API_KEY=...

# 3 · register → prepare → index → investigate
silentwitness register-evidence rocba ~/evidence/rocba.E01
silentwitness prepare rocba
silentwitness index rocba
silentwitness investigate rocba

# 4 · read the report + score against ground truth
cat cases/rocba/report.md
python -m harness.score_case --case cases/rocba --dataset rocba
Tip

You don't need to install Python or forensic tools by hand — install.sh does all of it. See Installation for requirements and model setup.

Core concepts / How it works

How it works

Evidence is parsed once into a searchable index. The agent only ever queries that index — it never free-reads raw evidence — and everything it records passes through verification gates before it can become a finding.

Parse once, query forever

A disk image holds millions of records; reading it top-to-bottom is hopeless. So the heavy work happens once, offline: prepare extracts artifacts read-only, nine targeted parsers plus a Sigma engine turn them into a per-case FTS5 index, and every indexed row carries a sha256 and an audit_id. At investigation time the agent runs only fast, bounded queries. See The evidence index.

The investigation loop

The agent works like a senior analyst: it starts from the staged Sigma detections, forms one hypothesis at a time, delegates to a memory / disk / network / log specialist, and records each finding as an observation (the cited bytes) plus an interpretation (what it means, a confidence, and what would change it). Full detail in Gates & coverage.

Two ways it stays honest

  • It can't fabricate. The citation and entity gates reject any claim that isn't grounded in cited evidence — see the firewall.
  • It can't stop early. A coverage gate blocks finalization until all five Key Questions are answered, and a live critic challenges weak findings mid-run.
Core concepts / Architecture

Architecture

Four parts and two externals. Eight trust boundaries separate the model from the evidence; six of them are enforced in code, not in prompts.

architectural · enforced in codesupplementary · prompt-based

Examiner — the CLI arch

The silentwitness command drives the case: register-evidence → prepare → index → investigate → review → export.

Evidence volume — read-only arch

/evidence is mounted ro,noexec,nosuid and validated fail-closed. It is a trust boundary the agent never crosses — the agent process never opens raw evidence at all.

silentwitness-mcp — the MCP server arch

The product. It holds the agent's only discovery surface (the index query tools), the adversarial sanitizer, the citation and entity gates, the hash-chained audit logger, and the HMAC-signed approval ledger. See the tool catalog.

silentwitness-agent — the reference agent arch

A Pydantic AI investigator: the hypothesis loop, the coverage-gate output_validator, the live critic with model-free contradiction detectors, and the Markdown report.

External supp

The optional claude-code config (CLAUDE.md + settings.json) and the model-agnostic LLM provider. These are prompt-based and supplementary — removing them degrades quality but unlocks no hallucinations, because enforcement lives at the server boundary.

Core concepts / The evidence index

The evidence index

The one-time, offline spine that turns raw artifacts into a fast, tamper-evident, searchable index — the only thing the agent ever reads.

Extract, read-only

prepare uses dfVFS to open the image read-only and copy out the high-value Windows artifacts: $MFT, $UsnJrnl, EVTX, registry hives, SRUM, Amcache, Prefetch, LNK, jumplists, and PowerShell transcripts.

Nine targeted parsers

Each artifact type has a dedicated feeder; together they normalize everything into citable index rows.

evtxmftregistrysrumusnjrnllnkjumplistprefetchpstranscript

Sigma detections

A pySigma engine evaluates DFIR rules over the parsed events and stages sigma:<level> alerts the agent starts from — failed-logon 4625, explicit-credential logon 4648, RDP interactive logon 4624, RDP outbound mstsc, suspicious PowerShell scriptblock 4104, and cloud-exfil tool execution.

FTS5 index, provenance per row

Everything lands in a per-case FTS5 index.db (~2.67M records on ROCBA). Every row carries a sha256 of its source artifact and the audit_id of the extraction that produced it — which is what makes any later citation verifiable.

Guardrails / The hallucination firewall

The hallucination firewall

AI hallucinates — we made that structurally hard. Ten places a claim has to survive: a raw, unverified claim enters at the top; only a grounded, cited finding leaves at the bottom.

01
No raw-evidence surface arch
The agent's only discovery interface is the index query tools — no registered tool writes to /evidence.
02
Typed tool vocabulary arch
27 Pydantic-typed tools, no execute_shell(); the registry refuses unregistered evidence paths.

Strips XML-role and chat-format tokens, injection regex, and Trojan-Source Unicode from evidence text.

The quoted span_text must be a verbatim substring of the cited record.
05

Guardrails / Gates & coverage

Gates & coverage

The three architectural gates that make a fabricated or incomplete finding impossible to record — and the self-correction loop that runs on top of them.

Citation gate

Inside record_observation, before anything is persisted: load the cited record, normalize it, and verify the quoted span_text is a verbatim substring. A mismatch is rejected with SPAN_NOT_IN_RECORD (or RECORD_NOT_FOUND). A fabricated quote cannot pass.

Entity gate

Then a spaCy en_core_web_lg NER pass plus DFIR regex (IPs, hashes, registry keys, Windows paths, accounts) extracts every named entity. Each must appear in a cited span, or the observation is rejected with HALLUCINATED_ENTITIES — and the response tells the agent exactly how to fix it.

The coverage gate

The agent's output_validator maps recorded observations to five Key Questions. Try to finalize with one unanswered and it raises ModelRetry, naming the gap and the artifacts that bear on it. The signals are generic DFIR concepts, so it generalizes across Windows data-theft cases — not just the image it was tuned on.

QuestionOn the ROCBA case
Q1 accounts / projects accessedSRL-Projects: Airwolf · Gunstar · Blue Thunder
Q2 what data was takenStark Research Labs R&D files + Key Data folder
Q3 where it was transferredG:\My Drive…Exported-PST (Google Drive) · OneDrive · Dropbox
Q4 how access was obtainedRDP LogonType 10 · BACKUPADMIN brute force · SDelete
Q5 when it occurredestablished intrusion window (timeline)

Self-correction

A live critic (separate agent, fresh context) challenges weak findings; beneath it, three model-free detectors run on every fire as a deterministic floor — execution-overclaim (presence ≠ execution), ghost-process (a live-process claim with no memory artifact), and timestamp-paradox (a time absent from the cited bytes). In the headline run the critic challenged 3 of 7 findings and the agent revised them.

Without the coverage gate even a capable model collapsed onto the single loudest signal (~540k brute-force detections) and scored below baseline. The gate restored breadth; a capable model then recalled all ten. Single run; recall varies by model and run-to-run — with gpt-5.2 + the gate, two runs scored 50% and 30%, yet both covered all five questions.

Known issues we are not hiding

  • Untrusted-marker leakage — the model copied [UNTRUSTED EVIDENCE] display markers into some prose. Does not affect recall or citations.
  • Single-run 100% ≠ stability — we report the distribution, not the peak.
  • Memory not yet in the recall path — current recall is disk-derived; disk⊕memory corroboration is planned.
  • Plaso contributes 0 events — libevtx crashes on the ROCBA EVTX; we route around it with targeted parsers and say so.
