## Per-claim verdicts

### Claim 1 — EZ Tools ship pre-installed on SIFT 2026 via dotnet 9 SDK

**VERDICT: TRUE in principle. PATH ASSUMPTION IS WRONG for 11 of 15 tools.**

`sift/scripts/zimmerman.sls` line 9 installs exactly these 15 tools:
`AmcacheParser, AppCompatCacheParser, bstrings, EvtxECmd, iisGeolocate, JLECmd, LECmd, MFTECmd, RBCmd, RecentFileCacheParser, RECmd, rla, SBECmd, SQLECmd, WxTCmd`.

dotnet 9 SDK comes from `sift/packages/dotnet.sls` (`dotnet-sdk-9.0` via `ppa:dotnet/backports`). dotnet binary lands at `/usr/bin/dotnet`. ✓

**The wrapper at `/usr/local/bin/<Tool>` is a bash script generated by `zimmerman.sls` lines 28-43. The wrapper has THREE branches:**

```jinja
{% if tool|lower == "iisgeolocate" or tool|lower == "recmd" or tool|lower == "sqlecmd" %}

dotnet /opt/zimmermantools/{{ tool }}.dll ${*}                  # FLAT - majority case
```

### Claim 2 — Invocation pattern `dotnet /opt/zimmermantools/<Tool>/<Tool>.dll <flags>`

**VERDICT: PARTIALLY WRONG.** The `dotnet <dll> <flags>` pattern is canonical (SANS blog April 2025 confirms `alias mftecmd='dotnet /opt/MFTEcmd/MFTECmd.dll'`). But the `<Tool>/<Tool>.dll` nested path is wrong for 11 of 15 tools (see Claim 1).

The correct argv is:
```python
DOTNET = "/usr/bin/dotnet"
DLL = "/opt/zimmermantools/MFTECmd.dll"        # FLAT
# or for EvtxECmd:
DLL = "/opt/zimmermantools/EvtxeCmd/EvtxECmd.dll"  # quirky case
# or RECmd/SQLECmd/iisGeolocate:
DLL = "/opt/zimmermantools/RECmd/RECmd.dll"    # NESTED
argv = [DOTNET, DLL, "-f", str(mft_path), "--csv", str(out_dir)]
```

**No known dotnet-9-on-Linux incompatibilities** for the 11 tools we plan to wrap. SANS blog (Apr 2025) covers MFTECmd, EvtxECmd, RECmd, PECmd, JLECmd, LECmd — all working. The 2026 net9 builds verified pulled cleanly.

### Claim 3 — EvtxECmd inner-directory quirk

**VERDICT: CONFIRMED. Spec is RIGHT on this single detail.**

`EvtxECmd.zip` (verified by `unzip -l` 2026-06-03) packs everything under top-level `EvtxeCmd/` (lowercase `e` after `Evtx`, capital E in `Cmd` would be wrong — confirmed `EvtxeCmd/Maps/`, `EvtxeCmd/EvtxECmd.dll`). Story `story-parse-evtx.md` line 24 and line 141 ARE correct on this path.

### Claim 5 — CSV / JSON output mode flags

**VERDICT: TRUE — all major tools support `--csv <out_dir>` + `--json <out_dir>` independently.** Both can be passed simultaneously and both outputs get written (verified by `MFTECmd/Program.cs` lines 403-413 — independent `if csv.IsNullOrEmpty() == false` checks). `--csvf <filename>` overrides the auto-generated CSV filename. `--jsonf` does the same for JSON.

**Flag syntax for our wrapped tools (verified):**
- `MFTECmd -f <mft> --csv <out_dir>`  (`--csvf` optional name override)
- `AmcacheParser -f <hive> --csv <out_dir>` (`-i` includes InventoryApplicationFile; produces MULTIPLE CSVs — see schema audit)
- `AppCompatCacheParser -f <hive> --csv <out_dir> [-c <controlSet>]`
- `PECmd -f <pf> --csv <out_dir>` OR `-d <prefetch_dir> --csv <out_dir>`
- `SBECmd -f <hive> --csv <out_dir>` OR `-d <hive_dir> --csv <out_dir>`
- `EvtxECmd -f <evtx> --csv <out_dir>` (filter via `--inc <eids>` / `--exc <eids>`; `--fj` = full JSON one-event-per-line; NO `--channel` filter — channels filter only by EID list)

### Claim 6 — Error handling

**VERDICT: SPEC ASSUMPTION IS WRONG. EZ Tools exit 0 on most errors.**

Source-counted in master:
- EvtxECmd Program.cs: 7 calls to `Environment.Exit(0)`, 0 calls to `Environment.Exit(1|-1)`
- MFTECmd Program.cs: 0 calls to `Environment.Exit(0)`, 6 calls to `Environment.Exit(1|-1)` — MFTECmd is the EXCEPTION (it does use non-zero exit)

This means our `TOOL_FAILED` advisory cannot key on exit code alone for EvtxECmd, PECmd, SBECmd, AmcacheParser, AppCompatCacheParser. We MUST parse stderr for Serilog level markers:
- `[ERR]` — `Log.Error(…)` rendered output
- `[FTL]` — `Log.Fatal(…)` rendered output

Both EZ Tools and Serilog use the default console sink which writes to **stdout** by default, not stderr — but for the tools in question (per Program.cs review) errors go to a `Console.Error`-bound sink (Serilog `WriteTo.Console(standardErrorFromLevel: LogEventLevel.Warning)` pattern is common but not universal). Defensive plan: capture BOTH stdout and stderr, scan for `[ERR]` / `[FTL]` / `"Error"` / `"Fatal"` strings AND exit code.

This is a NEW failure mode the specs need to handle.
